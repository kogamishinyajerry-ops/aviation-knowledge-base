- `part_of`: subject = part, object = whole. (e.g. "FADEC system part_of Boeing 737 MAX 8" → subject=FADEC, object=B737MAX8)
- `supersedes`: subject = newer/active, object = older/superseded. (Per `tests/fixtures/valid/relations/supersedes_far-clause-chain.yaml`.)
- `cites`: subject = citing entity, object = cited Document.
- `used_in`: subject = thing being used, object = context using it. (TurbulenceModel used_in SimulationCase → subject=k-omega-sst, object=naca0012-re6e6-aoa-sweep)
- `mitigated_by`: subject = failure/risk, object = mitigation. (FailureMode mitigated_by Procedure → subject=cpu-lockup, object=power-up-self-test)

## 3. Confidentiality Gating (DOC-04 — HARD RULE)

`restricted` and `itar_ear` documents are **NOT ingested into RAGFlow by default**.

The Phase-5 RAGFlow exporter (`scripts/exporters/to_ragflow.py`) reads
`metadata.yaml` for every doc and:
- `public` / `internal` → uploaded
- `restricted` / `itar_ear` → SKIPPED with a stderr warning

Until that exporter ships (Phase 5), human reviewers enforce the rule
manually: **do not put `restricted` / `itar_ear` documents into `docs/`
unless your repo is itself classified at the matching level.**

## 4. Manual Import Workflow

1. Author creates `docs/<domain>/<doc-id>/source.{pdf,md,…}` (PDFs go through Git LFS — see `.gitattributes`).
2. Author runs pandoc / OpenDataLoader to produce `processed.md` (or hand-writes it for short text-only docs).
3. Author writes `metadata.yaml` populating ALL 11 mandatory fields.
4. Author authors the matching `instances/entities/document/<doc-id>.yaml` (Document entity record) so other records can `cites` it.
5. Author opens PR; pre-commit + CI run `python scripts/validate.py` and `check-jsonschema`; PR cannot merge if either fails.
6. Reviewer (default: jane-reviewer for this demo, real curator team in production) signs off.

## 5. Scripted (AI-Assisted) Import Workflow

When an AI tool (Claude / Codex / DeepSeek / GPT) extracts entity or relation
records from a source document:

1. AI tool emits YAML with `provenance.method: "ai_extracted"`,
   `confidence.score` + `confidence.rationale`,
   structured `source.{document_id, locator, retrieval}`.
2. YAML lands at `instances/_pending/<entities|relations>/<type>/<slug>.yaml` —
   NEVER directly in canonical `instances/entities/` or `instances/relations/`.
3. Human reviewer reads, edits, sets:
   - `provenance.method`: change from `ai_extracted` → `hybrid_reviewed`
   - `provenance.reviewer`: a Person URI
   - `provenance.reviewed_at`: ISO timestamp
4. Reviewer opens PR moving the file from `_pending/` to canonical.
5. CI validates the promoted record against the H-Darrieus REJECT and the
   `_pending` promotion gate (per `instances/_pending/README.md`).

A worked example sits in `instances/_pending/entities/expert-note/pending-ai-extracted-fadec-thermal-margins.yaml`
(Phase 4 DEMO-06).

## 6. The H-Darrieus REJECT Rule (PROV-04)

Even a record that stays in `_pending/` is REJECTED by the validator if:

- `provenance.method == "ai_extracted"` AND
- `confidence.score > 0.85` AND
- `provenance.reviewer` is null or empty

Rationale (lifted from `instances/_pending/README.md`): high confidence on
unreviewed AI output is the exact failure mode that produced the H-Darrieus
fabricated-figure incident in the project history. We refuse to be that.

If you genuinely have AI-extracted output you trust at score > 0.85, you must
either (a) lower the score honestly, or (b) review it and bump method to
`hybrid_reviewed`. There is no third option.
