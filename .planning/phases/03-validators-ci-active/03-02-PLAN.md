<objective>
Build the **invalid fixture corpus** — one fixture per failure mode the Phase 3 validators must catch. Each fixture isolates exactly one rule violation so Wave-3 pytest tests can assert that the right validator catches the right error (no false positives, no missed detections).

Purpose: this is the truth set for "what does an invalid record look like?". Without it, the Wave-2 validators have no observable target and Wave-3 pytest has nothing to assert against. The directory layout (`invalid/<rule-name>/<descriptive-filename>.yaml`) makes the test parametrisation in Wave 3 trivial — pytest just walks the tree and asserts `rule==<dir-name>` errors are returned.

Output: 12 invalid YAML fixtures + 1 README.md documenting the fixture-to-rule mapping.
</objective>

Each fixture is a YAML record that is **valid in every way EXCEPT** the one rule it is testing. This is the most subtle requirement: if a fixture violates two rules, Wave-3 pytest cannot tell which validator caught what.

**Fixture 1 — `missing-provenance/expert-note_no-provenance.yaml`** (schema rule):
- Valid ExpertNote BUT delete the entire `provenance:` block
- Expected: `schema.validate_record` returns ≥1 ValidationError with rule="schema", message containing "provenance"

**Fixture 2 — `missing-schema-version/aircraft-model_no-schema-version.yaml`** (schema rule + cross-record VER-03):
- Valid AircraftModel BUT no `schema_version:` field
- Expected: `schema.validate_record` catches this via the entity.base required list

**Fixture 3 — `bad-uri-format/aircraft-model_bad-uri.yaml`** (ids rule, Wave-2 owner):
- Valid AircraftModel BUT `id: "not-a-valid-uri"` (no aviationkb:// scheme)
- Schema layer also catches this (pattern fails) — that is FINE; both layers should agree
- Expected: schema rule + ids rule both fire; pytest asserts both

**Fixture 4 — `bad-internal-id/relation-cites_bad-id.yaml`** (ids rule):
- Valid `cites` relation BUT add an extra field with wrong-shape internal id, e.g. `legacy_internal_id: "BadID_With_Underscores"` — note: this requires the schema to define `legacy_internal_id` as `$ref: $defs/internalId`. Since none of the current schemas do, simulate the failure by setting `subject: "comp:Bad_Underscore_ID"` (subject schema is `$defs/uri`, so this fails URI pattern). Use this same fixture to test BOTH the URI pattern and the ids validator's URI-format rule
- Expected: schema rule + ids rule fire

**Fixture 5 — `h-darrieus-rejected/expert-note_ai-extracted-high-conf-no-reviewer.yaml`** (provenance rule, PROV-04):
- Valid ExpertNote with:
  - `provenance.method: ai_extracted`
  - `confidence.score: 0.92`
  - NO `provenance.reviewer` key at all
- Expected: schema validation PASSES (ai_extracted is allowed by enum, reviewer is not schema-required)
- Expected: `provenance.validate_record` returns ValidationError(rule="provenance.h-darrieus", ...)

**Fixture 6 — `h-darrieus-empty-reviewer/expert-note_high-conf-empty-string-reviewer.yaml`** (provenance rule, PROV-04 edge case):
- Like Fixture 5 BUT `provenance.reviewer: ""` (empty string)
- The schema may or may not reject the empty string (depends on URI pattern strictness — reviewer uses `$defs/uri` which has `pattern: ^aviationkb://...`, so an empty string SHOULD fail schema validation too). Validate which is true and document the dual-fire.
- This fixture exists to document the H-Darrieus rule must treat empty-string reviewer as "no reviewer"

**Fixture 7 — `pending-not-hybrid-reviewed/expert-note_pending-still-ai-extracted.yaml`** (provenance rule, PROV-05):
