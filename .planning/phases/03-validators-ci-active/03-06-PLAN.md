# scripts/validators/ — Aviation KB Ontology Validators

> AI 接力开发指南: This directory implements the runtime rules that Phase 2
> schemas only documented. Read this file first if you are adding a new
> rule, debugging a CI failure, or extending the validator package.

## Overview

Five Python modules + one master CLI:

| Module | Rule names | What it catches |
|---|---|---|
| `schema.py` | `schema` | JSON Schema Draft 2020-12 violations against the type's leaf schema (composes _meta.schema.json) |
| `ids.py` | `ids.uri-format`, `ids.type-prefix-mismatch` | URI/internal-ID pattern + type-segment-vs-record-type mismatch (D-23/D-24/D-25) |
| `provenance.py` | `provenance.h-darrieus`, `provenance.pending-gate`, `provenance.schema-version-mismatch` | PROV-04 H-Darrieus REJECT, PROV-05 _pending promotion gate, VER-03 cross-record schema_version |
| `relations.py` | `relations.subject-not-found`, `relations.object-not-found`, `relations.supersession-incomplete`, `relations.supersession-target-not-found` | Cross-record subject/object resolution + supersession chain integrity |
| `links.py` | `links.broken-source-ref`, `links.source-not-document` | PROV-06 source.document_id resolves to a Document entity |

Master CLI: `python scripts/validate.py [PATH ...]` (default: `instances/`).

## Public API (stable)

```python
from scripts.validators.errors import ValidationError
# ValidationError is a frozen dataclass with: rule, severity, file, message, pointer

# Each validator module exposes:
def validate_record(path: Path, record: Any, **ctx) -> list[ValidationError]
# ctx provides: records, by_id (corpus index), repo_root
```

This signature is FROZEN. Phase 4+ extensions add new rules in NEW modules
or extend existing ones — they MUST NOT change `validate_record`'s shape
without a Phase-level ADR.

## How to Add a New Rule

1. Decide the right module (or create a new one):
   - JSON Schema-expressible? Add to the schema. No new validator needed.
   - Cross-field within one record? Add to `provenance.py` or create a new
     module if it does not fit the existing semantic groups.
   - Cross-record (needs the corpus)? Add to `relations.py` or `links.py`.
2. Create at least one VALID fixture (proves the green path) and at least
   one INVALID fixture (proves the red path) under
   `tests/fixtures/{valid,invalid}/<rule-name>/`.
3. Update `tests/fixtures/invalid/README.md` rule-to-fixture map.
4. Update `tests/test_validators.py` `_INVALID_DIR_TO_RULES` mapping.
5. Run `pytest -q` — must be green.
6. Run `python scripts/validate.py tests/fixtures/valid/` — must exit 0.

## CI

Three jobs run on every push and PR (`.github/workflows/ci.yml`):
1. `lint` — pre-commit on all files
2. `validate` — `python scripts/validate.py instances/` AND `tests/fixtures/valid/`
3. `test` — `pytest -q`

CI blocks merge if any job fails.

## Open Questions / Future Work

- **VER-03 hardening**: `provenance.schema-version-mismatch` is a WARNING in
  v0.1.0 because there is no v0.0.x corpus to reject. When `ontology/VERSION`
  bumps to 0.2.0+, harden to ERROR for any record whose schema_version is
  older than current minus one (the "N-1" rule).
- **Locale-aware error messages**: messages are English-only today; add ZH
  variants if/when contributor base shifts.
- **Performance**: full corpus scan is O(n) per record per validator. At
  10K+ records (post-MVP), profile and consider parallel validators.
