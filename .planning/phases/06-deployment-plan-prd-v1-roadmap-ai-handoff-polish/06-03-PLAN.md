## How to Read

Each entry has a uniform shape:

- **What** (one paragraph): what the feature is, what it does in user terms.
- **Why deferred** (one paragraph): why v1 explicitly excluded it.
- **Promote when** (a concrete, testable trigger): when this becomes true,
  the feature is eligible to enter the next ROADMAP.md phase pass.
- **v2 REQ-IDs** (links to REQUIREMENTS.md §v2): exact IDs this feature would close.
- **Pitfall guard** (link to PITFALLS.md if applicable): the pitfall this feature
  must NOT introduce while solving its problem.
- **Cross-references**: pointers to deploy docs / design docs that already
  partially scaffold this feature.

## How to Update

**Adding a new future feature**: copy the template, fill all 6 sections. Do NOT
add a feature without a "Promote when" trigger that is grounded in something
measurable (date, count, upstream bug closing, user demand, failure pattern).

**Firing a trigger**: when the trigger condition is met, file an ADR in
`.planning/decisions/` documenting the firing event, then propose a new ROADMAP
phase via `/gsd-roadmap-update`. Do NOT silently start work — the trigger
must be explicit and reviewable.

## 1. GraphRAG Layer

- **What**: Hybrid retrieval combining the existing vector + BM25 hybrid (Phase 5)
  with graph-traversal queries over relations. A user query like "list every
  RegulationClause that supersedes a clause cited by SimulationCase X" requires
  joining citation graph + supersession chain — graph traversal returns this
  in one hop, vector retrieval cannot.
- **Why deferred**: v1 scale (≤10K triples per CLAUDE.md Out-of-Scope) is below the
  break-even point where graph traversal beats hybrid vector + relational join.
  RAGFlow's hybrid retrieval already handles citation queries via metadata
  filters. Graph layer adds operational + storage complexity without proven win.
- **Promote when**:
  1. Total entity + relation instance count in `instances/` exceeds **10,000** records, OR
  2. ≥5 user queries in `evaluation/queries.yaml` are documented as failing
     because hybrid retrieval cannot follow a multi-hop relation chain
     (logged as `failure_mode: graph_traversal_needed` in eval results), OR
  3. A specific use case lands that requires path queries
     (e.g. "show all components affected by AD chain X→Y→Z" with cycle detection).
- **v2 REQ-IDs**: GRAPH-01, GRAPH-03, GRAPH-04
- **Pitfall guard**: PITFALLS.md Pitfall 11 (truth + cache discipline) — graph
  layer is a DERIVATIVE of YAML truth, not a parallel truth. Rebuildable from Git.
- **Cross-references**: `.planning/research/ARCHITECTURE.md` §"Migration Path to GraphRAG"; `scripts/exporters/to_jsonl_triples.py` (already produces triples).

## 2. Agent Layer

- **What**: Tool-using agents that operate on the knowledge base — guided
  ingestion (LLM proposes entity extractions, human approves before promotion),
  conflict detection (regulation X vs Y inconsistency surfacing), staleness
  detection (Regulation supersession alerts).
- **Why deferred**: Per CLAUDE.md Constraints + Out-of-Scope: "v1 = no decision-making agents,
  no auto-tool-use loops; v1 is RAG retrieval-augmented Q&A only." Agent
  introduction risks H-Darrieus failure-mode replay (Pitfall 2): unreviewed AI
  output entering canonical.
- **Promote when**:
  1. The `instances/_pending/` quarantine has accumulated ≥**100** AI-extracted
     records and the manual review queue is the bottleneck for ingestion velocity, AND
  2. Phase 4's `provenance.method = ai_extracted` reject-without-reviewer rule
     (PROV-04) has been measured stable for ≥3 months under real load (no false
     promotions slipping through), AND
  3. A specific failure-pattern justifies tool-use loops (e.g. "we keep missing
     supersession chain breaks because manual review can't keep up with regulation
     update frequency").
- **v2 REQ-IDs**: AGENT-01, AGENT-02, AGENT-03
- **Pitfall guard**: PITFALLS.md Pitfall 2 (AI extraction without review) — agents
  do NOT promote to canonical; they only stage in `_pending/` with confidence
  scores; human review remains the canonical promotion gate.
- **Cross-references**: `.planning/research/PITFALLS.md` Pitfall 2; `instances/_pending/README.md` (the staging convention this builds on).

## 3. Graph DB Backend (Neo4j / Nebula)

- **What**: Replace YAML-on-Git as the entity/relation truth source with a graph
  database providing native traversal, indexing, and concurrent multi-author writes.
- **Why deferred**: Per CLAUDE.md Constraints (no graph DB in v1) + STACK.md "What
  NOT to Use" — premature backend lock-in; YAML+JSON Schema sufficient for ≤10K
  triples; Git provides audit + diff for free.
- **Promote when**:
  1. GraphRAG Layer (entry §1 above) trigger has fired AND graph-on-vector hybrid
     measurably outperforms vector-only on ≥3 query categories, OR
  2. Concurrent multi-author writes become a bottleneck (≥3 reviewers blocked
     on PR queue ≥1 day, ≥1 week running) — YAML-on-Git serializes through PR review, AND
  3. ADR documents the migration path: choice (Neo4j vs Nebula vs ArangoDB),
     URI stability rule, rebuild-from-Git fallback.
- **v2 REQ-IDs**: GRAPH-02
- **Pitfall guard**: PITFALLS.md Pitfall 11 (truth + cache) — Git remains
  authoritative; graph DB is a synchronized derivative or replaces YAML only
  AFTER an explicit migration ADR. No silent dual-truth.
- **Cross-references**: `.planning/research/ARCHITECTURE.md` §"Migration Path to GraphRAG"; `.planning/research/STACK.md` §"What NOT to Use" (Neo4j/Nebula deferred rationale).

## 4. OCR Pipeline

- **What**: Optical Character Recognition for scanned-PDF source documents that
  lack a text layer. Activates RAGFlow's DeepDoc backend (or Tesseract /
  PaddleOCR fallback) instead of OpenDataLoader-PDF.
- **Why deferred**: Per CLAUDE.md Out-of-Scope: "Text-layer PDF/MD/HTML/DOCX only
  in v1; OCR adds non-deterministic fail mode." Phase 4 source documents (FAR
  CFD paper accident report) are all text-layer. OpenDataLoader-PDF handles 100%
  of the v1 corpus deterministically (~14× faster than DL-based parsers).
- **Promote when**:
  1. ≥**5** scanned-only PDF documents enter the corpus that OpenDataLoader-PDF
     cannot extract text from (verified by `metadata.yaml` flag `requires_ocr: true`), OR
  2. A user-priority document (regulatory amendment, accident report) is
     scanned-only and must be ingested for a specific deliverable.
- **v2 REQ-IDs**: OPS-04
- **Pitfall guard**: PITFALLS.md Pitfall 6 (chunking destroys tables) — OCR
  often loses table structure; the OCR pipeline MUST preserve table-as-atomic-chunk
  per RAG_PIPELINE.md §2.2.
- **Cross-references**: `.planning/research/STACK.md` §"Document Ingestion Stack" (DeepDoc fallback ready); `.planning/design/RAG_PIPELINE.md` §2.

## 5. Multi-tenant RBAC

- **What**: Role-based access control across multiple organizations / projects,
  with per-tenant isolation of `instances/`, `docs/`, and Wiki.js page trees.
- **Why deferred**: Per CLAUDE.md Out-of-Scope: "v1 = single-org admin/reader". Adding
  multi-tenant from day one over-designs the data model (every entity needs a
  `tenant_id` field, every validator needs tenant scoping, every query needs
  tenant filter). Premature.
- **Promote when**:
  1. ≥**2** distinct organizations or projects need to share the deployment
     while keeping their `instances/` content separate, AND
  2. Single-org admin/reader auth is documented as insufficient (specific
     access-control failure logged), AND
  3. ADR documents the chosen RBAC model (org-per-tenant vs project-per-tenant
     vs row-level), the tenant_id schema migration plan, and the
     query-scoping pattern.
- **v2 REQ-IDs**: OPS-02
- **Pitfall guard**: PITFALLS.md Pitfall 4 (schema migration) — adding `tenant_id`
  is a breaking schema change requiring full migration script + dual-version
  reader window.
- **Cross-references**: `ontology/_meta.schema.json` (where `tenant_id` would land); `deploy/authentik-phase2.md` (auth is prerequisite for RBAC).

## 6. SSO Unification (Authentik OIDC)

- **What**: Single sign-on across Wiki.js + RAGFlow via Authentik as OIDC IdP.
  Eliminates duplicate user accounts; centralizes auth audit.
- **Why deferred**: Per `deploy/authentik-phase2.md` — RAGFlow OIDC has open bug
  #12568 (Keycloak redirect-loop since Quart migration) and FR #3495 still pending.
  Wiki.js side is documented and works; RAGFlow side is the blocker. v1 = local
  accounts on each service.
- **Promote when**:
  1. RAGFlow issue #12568 is **closed** upstream (verify via GitHub issue API), AND
  2. RAGFlow FR #3495 is merged (OIDC support landed in a stable RAGFlow release ≥0.26), AND
  3. ≥**2** confirmed user accounts need cross-service navigation (logged as
     a real demand, not speculative).
- **v2 REQ-IDs**: OPS-01
- **Pitfall guard**: none specific (auth is orthogonal to data pitfalls).
- **Cross-references**: `deploy/authentik-phase2.md` (the authoritative deferral doc, with the same Promote-when conditions); `deploy/docker-compose.yml.draft` (commented authentik service block ready to uncomment when this fires).

## 7. Decision Agent

- **What**: An agent that takes a user goal (e.g. "draft a certification compliance
  matrix for Aircraft X against EASA CS-25") and autonomously navigates the KB,
  proposes draft outputs, asks clarifying questions, and revises based on feedback.
- **Why deferred**: Per CLAUDE.md Out-of-Scope: "Decision-making agents (auto-tool-use
  loops) excluded; v1 = RAG retrieval-augmented Q&A only." Decision agents
  produce contestable outputs ("the agent decided X") that are HARDER to audit
  than retrieval ("the system retrieved chunks Y, the LLM cited them"). Core Value
  ("every AI answer has citations") is structurally weaker for decision agents
  unless every action is provenance-tagged.
- **Promote when**:
  1. RAG Q&A (v1 deliverable) is measured **stable** in production for ≥6 months
     with citation-validation failure rate ≤1%, AND
  2. A specific deliverable use case is documented (e.g. "compliance matrix
     drafting") that RAG Q&A cannot satisfy because it requires multi-step
     synthesis with intermediate document creation, AND
  3. ADR specifies how decision-agent provenance is structured (every agent action
     produces an `ai_extracted` _pending entity with full audit trail), AND
  4. Agent Layer (entry §2 above) trigger has fired AND staging→canonical promotion
     pipeline has been load-tested with ≥**100** ai_extracted records reviewed
     without false promotion.
- **v2 REQ-IDs**: (none specific in REQUIREMENTS.md v2; this is a future-future
  category. Closure requires creating new REQ-IDs in a v2 requirements doc when
  the trigger fires.)
- **Pitfall guard**: PITFALLS.md Pitfall 2 (AI extraction without review) — agents
  STILL cannot promote to canonical without human review; the gate is permanent.
- **Cross-references**: `.planning/research/PITFALLS.md` Pitfall 2; CLAUDE.md
  Constraints "decision-making agents excluded" (this entry exists to make
  explicit when that constraint can be revisited).
